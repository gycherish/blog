> 如果你用过 NAS、服务器或虚拟化平台，大概率见过一个很神奇的能力：明明硬盘插在另一台机器上，自己的系统里却突然多出了一块“本地磁盘”。
> 这背后的常见技术之一，就是 iSCSI。

iSCSI 的核心思想并不复杂：它不是把远程目录共享给你，而是把远程存储伪装成一块可以分区、格式化、挂载的“裸磁盘”。操作系统以为自己在访问本地硬盘，实际读写请求会通过网络送到远端存储设备。

一句话概括：

**iSCSI 就是把 SCSI 磁盘命令封装进 TCP/IP，让网络另一头的存储看起来像本地硬盘。**

---

## 先从一个问题开始：为什么不用普通文件共享？

假设你有两台机器：

- 一台应用服务器，负责跑数据库、虚拟机或业务程序。
- 一台存储服务器，里面有很多硬盘，容量很大。

最直接的想法是：把存储服务器上的某个目录共享出来，让应用服务器通过 SMB 或 NFS 挂载。

这当然可以，但它共享的是“文件夹”。客户端看到的是远程目录，文件系统主要由服务端管理。

而有些场景需要的不是远程目录，而是一块真正意义上的“磁盘”：

- 数据库想独占一块数据盘。
- 虚拟化平台想把远程存储加入存储池。
- 系统想在远程块设备上创建 LVM、分区或文件系统。
- 应用需要直接面向块设备做 I/O。

这时候，iSCSI 就登场了。文件级共享和块级共享的差异，如下图：

![文件级存储和块级存储对比](iscsi-file-vs-block.svg)

## 文件级存储 vs 块级存储

理解 iSCSI，最重要的是先分清两类存储访问方式。

| 维度 | 文件级存储 | 块级存储 |
| --- | --- | --- |
| 常见协议 | SMB、NFS | SATA、SAS、FC、iSCSI、NVMe-oF |
| 客户端看到什么 | 目录和文件 | 一块磁盘或块设备 |
| 文件系统在哪里 | 通常由服务端管理 | 通常由客户端创建和管理 |
| 典型用途 | 文档共享、媒体库、团队文件夹 | 数据库、虚拟机磁盘、存储池 |
| 使用体验 | 像访问远程文件夹 | 像插入一块新硬盘 |

举个生活化的类比：

- SMB/NFS 像是别人把一个整理好的文件柜开放给你，你按目录拿文件。
- iSCSI 像是别人把一块还没装修的空地交给你，你自己决定怎么分区、铺地板、建房间。

iSCSI 属于后者。它暴露的是磁盘块，不是现成文件。

---

## iSCSI 到底是什么？

iSCSI 全称是 **Internet Small Computer System Interface**。

名字里有两个重点：

- **SCSI**：一种经典的存储命令体系，用来描述读块、写块、查询设备、刷新缓存等磁盘操作。
- **Internet**：这里不是说一定要走公网，而是说它可以跑在 IP 网络上，通常使用 TCP。

传统情况下，操作系统通过本机总线或专用存储网络向硬盘发送 SCSI 命令。iSCSI 做的事情是：

1. 客户端把磁盘读写变成 SCSI 命令。
2. iSCSI 把这些命令封装进 TCP/IP 数据包。
3. 数据包通过以太网发到远程存储。
4. 远程存储执行命令，把结果再通过网络返回。

这条链路可以概括成下图：

![iSCSI 分层链路](iscsi-layered-path.svg)

应用写入经过文件系统、块设备层和 iSCSI/TCP/IP，最终落到远端存储池。所以，从系统视角看，它像本地盘；从物理视角看，它其实在网络另一端。

---

## iSCSI 的几个核心角色

iSCSI 里有几个术语，一开始看会有点硬，但它们其实对应关系很直观。

| 术语 | 中文理解 | 作用 |
| --- | --- | --- |
| Initiator | 发起端 | 使用远程磁盘的一方，通常是服务器或客户端 |
| Target | 目标端 | 提供远程磁盘的一方，通常是 NAS、SAN 或存储服务器 |
| LUN | 逻辑单元号 | Target 暴露出来的一块逻辑磁盘 |
| Portal | 网络入口 | Target 的 IP 和端口，默认端口通常是 3260 |
| IQN | iSCSI 限定名 | Initiator 或 Target 的唯一标识 |
| CHAP | 认证机制 | 用账号密码验证连接方身份 |

可以把它想成一次“远程插硬盘”的关系，如下图：

![Initiator、Target 和 LUN 的关系](iscsi-initiator-target-lun.svg)

其中 Initiator 负责发起连接，Target 负责提供存储服务，LUN 就是最终出现在客户端系统里的那块“盘”。

## 为什么系统真的会把它当本地盘？

因为 iSCSI 暴露给操作系统的是块设备。在 Linux 上，它可能表现为：

```text
/dev/sdb
/dev/sdc
```

在 Windows 上，它会出现在“磁盘管理”里，像一块新插入的磁盘一样，需要初始化、分区、格式化。这意味着你可以像操作普通硬盘一样操作它：

- 创建分区。
- 格式化为 ext4、xfs、NTFS 等文件系统。
- 挂载到某个目录或盘符。
- 放数据库数据文件。
- 做 LVM、RAID 上层组合或虚拟机磁盘。
- 加入虚拟化平台的存储池。

但请记住：**像本地盘，不等于就是本地盘。**

它仍然依赖网络、交换机、存储端服务和后端磁盘。一旦链路抖动、Target 重启或网络中断，客户端看到的就是“磁盘 I/O 出问题”。

## iSCSI、NFS、SMB 到底怎么选？

很多人第一次接触 iSCSI 时，会把它和 NAS 文件共享混在一起。下面这张表可以快速判断。

| 场景 | 更适合 iSCSI | 更适合 NFS / SMB |
| --- | --- | --- |
| 多人共享文档 | 否 | 是 |
| 家庭影视库 | 否 | 是 |
| 数据库独立数据盘 | 是 | 视情况 |
| 虚拟机磁盘存储 | 是 | NFS 也常见 |
| 需要客户端自己格式化 | 是 | 否 |
| 多台普通机器同时写同一份数据 | 不建议 | 是 |
| 需要像本地硬盘一样被系统识别 | 是 | 否 |

一个特别重要的提醒：

> 普通文件系统通常不能让多台机器同时挂载同一个 iSCSI LUN 并写入。
> 除非你使用的是集群文件系统，或者上层软件明确支持这种共享块设备模式，否则很容易造成数据损坏。

换句话说，iSCSI 不是“多人共享盘”的简单替代品。它更像是把一块远程硬盘“分配”给某台机器使用。

## iSCSI 常见使用场景

iSCSI 最常见的落点，是那些“需要块设备”的场景。

### 虚拟化平台

VMware、Hyper-V、Proxmox 等平台经常使用 iSCSI 作为共享存储。计算节点通过网络连接集中存储，虚拟机磁盘文件或数据块放在远端存储池中：

![虚拟化平台使用 iSCSI 共享存储](iscsi-virtualization-storage.svg)

### 数据库独立数据盘

数据库经常希望数据目录放在独立磁盘上，方便扩容、迁移、备份和性能隔离。iSCSI 可以提供一块远程块设备，让数据库服务器像使用本地数据盘一样使用它。

### 家用 NAS 和实验环境

很多 NAS 都支持创建 iSCSI LUN。你可以在 Windows 或 Linux 上连接它，把 NAS 上的一部分容量变成本机磁盘，用来做实验、备份、虚拟机测试盘等。

### 集中化存储

在中小规模机房里，iSCSI 可以让多台服务器使用集中存储资源，降低本地硬盘扩容和更换的复杂度。

---

## iSCSI 的优点

iSCSI 能流行起来，主要是因为它刚好站在“专业块存储”和“普通以太网”之间：

- **基于普通 IP 网络**：不一定需要昂贵的光纤通道设备，网络质量够好，普通以太网也能承载。
- **被系统识别为标准块设备**：大量现有工具能直接使用，对很多软件来说它就是一块磁盘。
- **容量集中管理**：后端统一管理磁盘池，再按需把 LUN 划分给不同服务器。
- **契合虚拟化与数据库**：这类系统天然面向块设备，iSCSI 正好提供了这层抽象。
- **支持多路径与冗余**：生产环境可用 MPIO、多网卡、多交换机提升可用性。

---

## iSCSI 的坑：它不是魔法

iSCSI 的体验很像本地盘，但它的稳定性和性能取决于整条链路。下图展示了这些风险如何沿链路传递：

![iSCSI 风险链路](iscsi-risk-chain.svg)

网络、控制器和后端磁盘池上的问题，都可能在客户端表现为磁盘 I/O 异常，常见风险包括：

- 网络断开会表现为磁盘 I/O 异常。
- 网络延迟会直接影响读写延迟。
- 交换机拥塞可能让应用突然卡住。
- Target 重启可能导致客户端文件系统报错。
- 多台机器误挂同一个 LUN 并写入，可能破坏文件系统。
- 没有访问控制时，错误客户端可能连到不该看的磁盘。

所以生产环境里使用 iSCSI，通常会有几条基本原则：

- 单独的存储网络或 VLAN。
- 稳定的交换机和网卡。
- CHAP 认证和访问控制。
- 必要时使用 MPIO。
- 不把 iSCSI 直接暴露到公网。

## 性能要看哪里？

iSCSI 的性能不是只看“协议快不快”，而是看整条路径的瓶颈。

![iSCSI 性能瓶颈链路](iscsi-performance-path.svg)

你需要关注三个指标：

| 指标 | 含义 | 受什么影响 |
| --- | --- | --- |
| 吞吐 | 每秒能传多少数据 | 网卡速率、交换机、后端磁盘顺序读写 |
| IOPS | 每秒能处理多少次 I/O | SSD/HDD、队列深度、控制器、随机读写能力 |
| 延迟 | 一次 I/O 多久返回 | 网络延迟、存储响应时间、拥塞情况 |

几个常见建议：

- 千兆网可以用，但别期待它承担高负载数据库或大量虚拟机。
- 万兆网会明显改善吞吐上限。
- SSD 后端对随机 I/O 帮助很大。
- Jumbo Frame 只有在端到端都正确配置时才可能有收益。
- MPIO 主要解决冗余和多路径利用，不是随便一开就线性翻倍。
- iSCSI 的上限通常由网络和后端存储共同决定。

## 安全上一定要注意

iSCSI 是块存储协议，一旦权限搞错，后果往往比普通文件共享更严重。因为客户端拿到的是整块盘，可能直接读写文件系统底层数据。

推荐的安全做法：

- 使用 CHAP 认证。
- 只允许指定 Initiator 访问指定 Target 或 LUN。
- 用防火墙限制 3260 端口访问来源。
- 放在独立 VLAN 或独立物理网络中。
- 不暴露到公网。
- 给不同业务划分不同 LUN，避免误操作影响其他系统。

一个更稳妥的安全边界可以参考下图：

![iSCSI 安全架构](iscsi-security-architecture.svg)

用存储 VLAN、防火墙、CHAP 和 ACL 把 iSCSI 访问范围限制在授权主机内。

一句话：**iSCSI 应该像内网基础设施一样被保护，而不是像普通网页服务一样对外开放。**

## 极简实践流程：从远程 LUN 到本地磁盘

不同系统和设备的界面不一样，但流程基本类似。

### 存储端

1. 在存储池里创建一个 LUN。
2. 创建或选择一个 iSCSI Target。
3. 把 LUN 映射到这个 Target。
4. 配置允许访问的 Initiator IQN。
5. 开启 CHAP 或其他认证。
6. 记录 Target 的 IP、端口和 IQN。

### 客户端

1. 安装或打开 iSCSI Initiator。
2. 发现 Target。
3. 登录 Target。
4. 系统识别出新磁盘。
5. 初始化磁盘。
6. 分区、格式化。
7. 挂载到目录或分配盘符。
8. 开始使用。

![从远程 LUN 到本地磁盘的实践流程](iscsi-practice-flow.svg)

在 Linux 上，你可能会看到类似这样的结果：

```text
$ lsblk
NAME   SIZE TYPE MOUNTPOINT
sda    100G disk
├─sda1 512M part /boot
└─sda2 99G  part /
sdb    500G disk
└─sdb1 500G part /data
```

这里的 `/dev/sdb` 可能就是来自远程 iSCSI Target 的 LUN。

## Windows 和 Linux 里会是什么样？

### Windows

Windows 内置了 iSCSI Initiator。连接 Target 后，可以在“磁盘管理”里看到一块新磁盘。接下来就和普通新硬盘一样：

1. 联机。
2. 初始化为 GPT 或 MBR。
3. 创建分区。
4. 格式化为 NTFS 或 ReFS。
5. 分配盘符。

### Linux

Linux 常用 `open-iscsi` 作为 Initiator。连接成功后，系统会出现新的块设备，例如 `/dev/sdb`。

常见操作包括：

```bash
lsblk
sudo fdisk /dev/sdb
sudo mkfs.xfs /dev/sdb1
sudo mkdir -p /data
sudo mount /dev/sdb1 /data
```

当然，生产环境还要考虑开机自动登录、自动挂载、超时参数、多路径配置等细节。

## 常见误区

- **“iSCSI 是文件共享”**：不是。它是块级存储，给客户端的是磁盘块，不是共享目录。
- **“多台机器可以随便同时挂一个 LUN”**：非常危险。普通文件系统不适合多主机同时写同一个块设备，除非用集群文件系统或平台提供的并发访问机制。
- **“用了 iSCSI 就一定比 NFS/SMB 快”**：不一定。性能取决于网络、后端磁盘、缓存、访问模式和协议实现。
- **“iSCSI 可以暴露到公网”**：强烈不建议。它是存储基础设施协议，不是面向公网的服务。
- **“远程盘断线没什么影响”**：影响可能很严重，相当于磁盘突然失联，轻则应用报错，重则文件系统损坏。

---

## 什么时候应该用 iSCSI？

适合用 iSCSI 的情况：

- 你需要的是块设备，而不是共享文件夹。
- 你希望客户端自己创建文件系统。
- 你在搭建虚拟化、数据库或集中块存储。
- 你有稳定的内网和可控的存储环境。

不太适合用 iSCSI 的情况：

- 只是想让多人共享文档。
- 只是想访问 NAS 上的电影、照片、备份文件。
- 网络环境不稳定。
- 不清楚多主机同时访问同一 LUN 的风险。
- 想把它作为公网存储服务。

## 结尾：iSCSI 的本质

iSCSI 听起来像一个很专业的存储协议，但它的底层逻辑可以拆得很清楚：

- 本地系统想读写磁盘。
- 读写请求被变成 SCSI 命令。
- SCSI 命令被封装进 TCP/IP。
- 网络把请求送到远端存储。
- 远端存储执行读写。
- 本地系统以为自己在操作一块本地盘。

所以，iSCSI 最迷人的地方就在于这个抽象：

**磁盘可以不在这台机器里，但块设备可以出现在这台机器上。**

用得好，它能让存储集中、容量灵活、虚拟化和数据库部署更方便。用不好，它也会把网络抖动、权限配置和多主机写入这些问题直接变成磁盘事故。

记住最后这句话就够了：

**需要远程文件夹，用 SMB 或 NFS；需要网络对面的“本地盘”，才轮到 iSCSI。**
