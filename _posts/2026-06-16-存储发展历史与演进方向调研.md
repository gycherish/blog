## 摘要

存储技术的发展不是一条单线替代史，而是一部围绕 **容量、性能、可靠性、成本、可管理性、数据访问模型** 不断重新平衡的历史。

从磁带、硬盘、RAID，到 DAS、NAS、SAN，再到对象存储、分布式存储、软件定义存储、NVMe-oF、CXL 和 AI 数据基础设施，存储系统一直在回答三个问题：

```text
怎么存得下？
怎么取得快？
怎么管得住？
```

今天的存储已经不只是“磁盘”或“阵列”，而是横跨介质、协议、网络、数据管理、云平台、AI 数据管线的一整套基础设施。

## 早期：从顺序访问到随机访问

早期数据存储主要依赖打孔卡、磁带等顺序访问介质。磁带容量相对可观，成本低，适合备份和归档，但随机读取效率很差。要读取某段数据，通常需要从前往后顺序扫描。

真正改变存储形态的是硬盘。

Computer History Museum 记录，IBM 在 1956 年向 Zellerbach Paper 交付了第一套商用硬盘系统 IBM Model 350，作为 IBM 305 RAMAC 的一部分。RAMAC 的关键价值不是容量本身，而是让数据可以随机访问。来源：[Computer History Museum - First commercial hard disk drive shipped](https://www.computerhistory.org/storageengine/first-commercial-hard-disk-drive-shipped/)

IBM 自己的历史页面也强调，IBM 305 RAMAC 是第一台使用随机访问磁盘驱动器的计算机，是后来硬盘体系的源头。来源：[IBM - RAMAC](https://www.ibm.com/history/ramac)

这个阶段的本质变化是：

```text
磁带：顺序访问，便宜，适合归档
硬盘：随机访问，昂贵但更适合在线业务
```

也就是说，存储第一次从“保存记录”走向“实时访问数据”。

## RAID：从单块磁盘到磁盘阵列

随着 CPU 和内存性能快速提升，磁盘 I/O 开始成为系统瓶颈。单块大盘虽然容量增加，但性能、可靠性和成本很难同时满足需求。

1988 年，Patterson、Gibson、Katz 在 SIGMOD 发表经典论文 **A Case for Redundant Arrays of Inexpensive Disks**，提出 RAID 思想，用多块便宜磁盘组成阵列，在性能、可靠性、功耗、扩展性上替代单块昂贵大盘。论文介绍了多个 RAID level，并把 RAID 与当时的大型磁盘方案对比。来源：[ACM DOI](https://dl.acm.org/doi/10.1145/50202.50214)、[Berkeley 技术报告](https://www2.eecs.berkeley.edu/Pubs/TechRpts/1987/5853.html)、[A Case for Redundant Arrays of Inexpensive Disks 论文 PDF](https://www.cs.cmu.edu/~garth/RAIDpaper/Patterson88.pdf)

值得一提的是，业界后来把 RAID 里的 “I” 从 **Inexpensive（廉价）** 重新解释为 **Independent（独立）**，因为重点逐渐从“便宜”转向“多块独立磁盘协同”。SNIA 词典今天就把 RAID 定义为 Redundant Array of Independent Disks。来源：[SNIA Dictionary - RAID](https://www.snia.org/education/online-dictionary/term/raid)

RAID 的思想非常重要：

- 把多块磁盘看成一个整体。
- 用条带化提升吞吐。
- 用镜像或校验提升可靠性。
- 用普通硬件组合出更强的存储能力。

这也是后来分布式存储的前奏：**用多个不完美组件，组合出更可靠的整体系统。**

不过校验型 RAID（RAID 5/6）也带来一个经典隐患——**写空洞**：数据块和校验块是分两次写的，若两次之间断电，校验和数据就不一致，可能在毫无察觉的情况下损坏整条条带。传统硬件 RAID 靠电池保护缓存来缓解；写空洞也是后来 ZFS 的 RAID-Z 试图规避的重要问题之一（见下一节）。来源：[OpenZFS - RAIDZ（消除 RAID-5 写空洞）](https://openzfs.github.io/openzfs-docs/Basic%20Concepts/RAIDZ.html)

## 本地文件系统与 ZFS：让文件系统“不信任磁盘”

RAID 解决的是“多块盘怎么组成一块可靠的盘”，而**文件系统**解决的是“一块盘上怎么组织文件”。这条单机主线常被忽略，但它直接影响数据完整性。

DOS/Windows 一脉是 FAT → NTFS（带日志、权限的 Windows 文件系统）；Unix/Linux 一脉是 ext2 → ext3 → ext4，其中 ext3 引入**日志**：先把“打算做的改动”记到日志，崩溃后即可重放或回滚，避免文件系统处于半更新的损坏状态。

真正的里程碑是 **ZFS**（最初叫 Zettabyte File System，源自 Sun，现以 OpenZFS 延续）。它把传统上分离的三层——**文件系统 + 卷管理 + 软件 RAID**——合并成一个整体，因此文件系统清楚每个数据块的物理位置。三个关键特性：

- **写时复制（CoW）**：永不就地覆盖，改动写到新位置、事务提交后再原子切换元数据指针，磁盘上的状态始终一致，崩溃后无需 `fsck`。
- **端到端校验和**：每个块的校验和存在父块指针里，读取时校验；若有镜像或 RAID-Z 副本，可**自动修复**（self-healing）静默损坏（bit rot）——这是传统 RAID 检测不到的。
- **RAID-Z 消除写空洞**：借助 CoW，每次写都是“整条带写到新位置、原子提交”，从设计上根除了上一节提到的写空洞。

来源：[OpenZFS - Checksums](https://openzfs.github.io/openzfs-docs/Basic%20Concepts/Checksums.html)、[Klara Systems - ZFS vs RAID](https://klarasystems.com/articles/zfs-vs-raid-zfs-redundancy-done-right/)

Linux 一侧的对应物是 **btrfs**（B-tree 文件系统），同样基于写时复制，提供校验和、快照与多设备 RAID。来源：[btrfs Introduction 文档](https://btrfs.readthedocs.io/en/latest/Introduction.html)

ZFS 的意义在于：它让“**文件系统主动怀疑磁盘**、用校验和兜住静默损坏”成为主流，并预演了后来分布式存储对“端到端数据完整性”的强调。

## DAS、NAS、SAN：存储开始网络化

![块、文件、对象三种存储访问模型](storage-access-models.svg)

早期服务器通常使用 DAS，磁盘直接连接在服务器上，简单、性能可控，但扩容、共享和集中管理都很困难。

企业系统规模扩大后，存储逐渐从服务器内部独立出来，出现 NAS 和 SAN 两条主线。

### NAS：文件共享

NAS 面向文件访问，客户端看到的是目录和文件。典型协议包括 NFS 和 SMB/CIFS。

NFS 经历了清晰的版本演进：初版见 RFC 1094（NFSv2，1989），NFSv3 见 RFC 1813（1995），NFSv4.0 原版见 RFC 3530（2003，后由 RFC 7530 更新）。其后两个版本尤其重要：**NFSv4.1（RFC 5661，2010）引入了 pNFS（并行 NFS），允许客户端并行直连多个数据服务器**，这正是高性能与 AI 场景看重的能力；**NFSv4.2（RFC 7862，2016）**进一步加入服务端复制、稀疏文件等特性。来源：[RFC 1094](https://www.rfc-editor.org/info/rfc1094)、[RFC 1813](https://datatracker.ietf.org/doc/html/rfc1813)、[RFC 3530](https://datatracker.ietf.org/doc/html/rfc3530)、[RFC 7530](https://datatracker.ietf.org/doc/html/rfc7530)、[RFC 5661](https://datatracker.ietf.org/doc/rfc5661/)、[RFC 7862](https://www.rfc-editor.org/rfc/rfc7862.html)

SMB 是 Windows 世界最核心的文件共享协议之一。Microsoft 文档定义 SMB 为网络文件共享协议，允许应用或用户通过网络读取、创建、更新远程服务器上的文件。严格说，CIFS（Common Internet File System）不是另一条独立路线，而是 SMB 的一个早期方言/版本名称；今天更常见、更准确的统称是 SMB，尤其是 SMB 2.x/3.x 之后。来源：[Microsoft - SMB overview](https://learn.microsoft.com/en-us/windows-server/storage/file-server/file-server-smb-overview)、[Microsoft - SMB Protocol and CIFS Protocol Overview](https://learn.microsoft.com/en-us/windows/win32/fileio/microsoft-smb-protocol-and-cifs-protocol-overview)

NAS 适合：团队共享目录、家庭 NAS、文档与媒体文件、多客户端文件访问。

### SAN：块设备共享

SAN 面向块设备访问。服务器看到的不是文件夹，而是一块远程磁盘或 LUN。典型技术包括 Fibre Channel、iSCSI、后来的 NVMe-oF。

iSCSI 的 RFC 3720 定义了在 TCP 之上传输 SCSI 的协议；后来的 RFC 7143 对 iSCSI 协议进行了整合定义。来源：[RFC 3720](https://datatracker.ietf.org/doc/html/rfc3720)、[RFC 7143](https://datatracker.ietf.org/doc/html/rfc7143)

SAN 适合：数据库数据盘、虚拟机磁盘、高性能块存储、企业集中存储阵列。

NAS 和 SAN 的差别可以简单理解为：

```text
NAS：共享文件
SAN：共享块设备
```

## 对象存储：云时代的默认数据底座

互联网和云计算让非结构化数据爆炸式增长。图片、视频、日志、备份、模型文件、数据湖对象，都不再适合简单放进传统目录树或块设备里管理。

对象存储的核心抽象是：

```text
Bucket / Container
Object
Key
Metadata
HTTP API
```

2006 年 3 月 14 日，AWS 发布 Amazon S3，新闻稿称其提供“高度可扩展、可靠、低延迟、低成本”的数据存储基础设施。来源：[Amazon Press Center](https://press.aboutamazon.com/2006/3/amazon-web-services-launches)、[AWS - Twenty years of Amazon S3](https://aws.amazon.com/blogs/aws/twenty-years-of-amazon-s3-and-building-whats-next/)

对象存储解决的不是“像磁盘一样快”，而是：海量扩展、跨地域复制、多租户、API 化访问、低成本保存非结构化数据、支撑数据湖和云原生应用。

OpenStack Swift 是对象存储的重要开源代表，用标准服务器集群提供冗余、可扩展的 PB 级存储。来源：[OpenStack Swift introduction](https://docs.openstack.org/swift/pike/admin/objectstorage-intro.html)。SNIA 的 CDMI 标准则试图为云存储访问和数据管理提供统一接口。来源：[SNIA CDMI](https://www.snia.org/cdmi)

### 纠删码：用更少空间换同等可靠性

对象存储和大规模分布式存储常在**冷数据、大对象或容量优化场景**中使用纠删码来兼顾可靠性与成本；而热数据、小对象或低延迟场景仍可能使用多副本。纠删码把对象切成 k 块数据，再算出 m 块校验，分散到不同节点；任意 k 块即可还原，可容忍 m 块同时损坏，主流算法是 Reed-Solomon。

它和“3 副本”的对比很直观：3 副本要 200% 冗余、只有 33% 可用空间；而 RS(10,4) 仅需 1.4 倍空间、可用率高得多，却仍能容忍 4 块失效。代价是**重建更贵**——补一块要从 k 个节点读数据并做解码运算，更耗网络和 CPU。因此实践中常见“热/小数据用副本、冷/大数据用纠删码”的分工。Ceph、HDFS 等系统都提供纠删码能力；以 HDFS 为例，其文档明确把 EC 描述为相对三副本更节省空间的冗余方式。许多云对象存储也会在冷/大对象场景采用类似的冗余策略。来源：[Ceph - Erasure code](https://docs.ceph.com/en/latest/rados/operations/erasure-code/)、[HDFS Erasure Coding](https://hadoop.apache.org/docs/current/hadoop-project-dist/hadoop-hdfs/HDFSErasureCoding.html)

## 分布式存储：从硬件阵列到普通服务器集群

![分布式存储从专用阵列到普通服务器集群](storage-distributed-architecture.svg)

云规模数据中心推动存储从专用硬件走向普通服务器集群。几个经典系统非常关键。

### GFS：面向大规模数据处理的分布式文件系统

Google 2003 年发表的 GFS 论文定义了一个面向大规模数据密集应用的分布式文件系统，在廉价普通硬件上提供容错能力，并为大量客户端提供高聚合性能。来源：[Google Research - GFS](https://research.google/pubs/the-google-file-system/)

关键思想：大文件优先、chunk 分片、master 管理元数据、chunkserver 保存数据、把普通硬件故障当成常态。

### Bigtable：大规模结构化数据

Google 2006 年发表 Bigtable 论文，将其定义为面向大规模结构化数据的分布式存储系统，可扩展到数千台普通服务器和 PB 级数据。来源：[Google Research - Bigtable](https://research.google/pubs/bigtable-a-distributed-storage-system-for-structured-data/)

Bigtable 代表了“存储不只是文件系统，还可以是大规模结构化数据服务”的方向。

### Dynamo：高可用键值存储

Amazon 2007 年发表 Dynamo 论文，描述了核心服务使用的高可用、可扩展键值存储；为了 always-on 体验，在某些故障场景下牺牲强一致性，并用对象版本和应用辅助冲突解决。来源：[Amazon Science - Dynamo](https://www.amazon.science/publications/dynamo-amazons-highly-available-key-value-store)

Dynamo 把“可用性、一致性、分区容忍、冲突解决”带入互联网规模存储设计。

### Ceph：统一的分布式存储底座

Ceph 2006 年 OSDI 论文把它定义为可扩展、高性能、可靠的分布式系统，并用 CRUSH 这类伪随机分布函数减少集中式分配表。来源：[USENIX - Ceph](https://www.usenix.org/conference/osdi-06/ceph-scalable-high-performance-distributed-file-system)

现代 Ceph 一套底座同时提供 RADOS 对象层、RBD 块设备、CephFS 文件系统和 S3/Swift 兼容对象网关。Red Hat 把它定位为面向私有云的软件定义存储平台。来源：[Red Hat Ceph Storage](https://www.redhat.com/en/technologies/storage/ceph)

这类系统说明存储开始从“买一台阵列”转向“用软件组织很多普通服务器”。

## 数据缩减与存储效率：精简配置、去重、压缩

在“怎么存得下、还要存得省”这条线上，企业存储普遍叠加三项能力：

- **精简配置（thin provisioning）**：按实际写入量分配物理空间，主机看到 1TB 卷、阵列只消耗真正用掉的容量。提升利用率，但需要监控避免超额订阅后真实空间耗尽。
- **去重（deduplication）**：相同的数据块只存一份，重复处用指针引用，对备份、虚拟机镜像等高重复场景极有效；分**在线**（写入路径上先去重再落盘）和**后处理**（先落盘、后台再去重）两种，前者省容量但增加写延迟，后者写得快但需要临时落地空间。
- **压缩（compression）**：在块内做编码压缩，对已压缩/已加密数据收益很小。

来源：[Pure Storage - thin provisioning](https://blog.purestorage.com/purely-technical/understanding-thin-provisioning-does-it-reduce-storage-capacity/)、[StarWind - dedup vs compression](https://www.starwindsoftware.com/blog/what-is-data-reduction-deduplication-vs-compression/)

## 超融合（HCI）：把存储塞回服务器

网络化存储把存储从服务器里独立出去，而**超融合（HCI）**又把它“塞回”服务器——只不过这次是用软件。HCI 把计算和存储放在同一批通用 x86 节点上，每个节点贡献自己的 CPU、内存和本地盘，再由一层 SDS 把所有本地盘虚拟成一个共享、冗余的存储池，替代独立的外置阵列；扩容通常以节点为单位，也可能按产品能力区分通用节点、计算密集节点或存储密集节点。

它与传统 SAN 的“计算 / 网络 / 专用阵列”三层架构形成对比：在部分场景下能降低初始部署和扩容的复杂度、按节点平滑扩展。但**是否更省成本要看规模、软件授权和计算/存储配比**——当存储与计算比例失衡、授权昂贵或扩容粒度受限时，HCI 未必更便宜。代表是 VMware vSAN（纯软件，跑在你选的硬件上）和 Nutanix（软硬件一体的一体机）。来源：[Nutanix - HCI](https://www.nutanix.com/hyperconverged-infrastructure)、[Computer Weekly - HCI vs SAN](https://www.computerweekly.com/feature/Hyper-converged-infrastructure-vs-NAS-SAN-shared-storage)

## SSD、NVMe 与 NVMe-oF：性能路径的重构

HDD 的机械结构决定了随机访问延迟很难继续大幅下降。SSD 的出现把瓶颈从机械寻道转移到接口、协议栈和软件路径。

SATA/SAS 来自更早的磁盘时代，而 NVMe 是为 PCIe SSD 设计的新接口，从一开始就为了定义主机软件如何通过 PCIe 与非易失性内存通信，提供比旧接口更低延迟、更可扩展的能力。来源：[NVM Express Base Specification](https://nvmexpress.org/specification/nvm-express-base-specification/)

NVMe-oF 则把 NVMe 从本地 PCIe 扩展到网络。NVMe-oF 1.0 于 2016 年 6 月发布，把 NVMe 扩展到 Ethernet、Fibre Channel、InfiniBand、RDMA 等传输；NVMe-oF 1.1 于 2019 年加入更细粒度 I/O 资源管理、端到端流控、NVMe/TCP 等能力。来源：[NVM Express - NVMe-oF](https://nvmexpress.org/specification/nvme-of-specification/)

2021 年的 **NVMe 2.0** 还做了一次重要的**模块化重构**：把规范拆成 Base 基础规范 + 命令集（NVM、ZNS、KV）+ 传输（PCIe、FC、RDMA、TCP）+ 管理接口的“规范库”，让本地与网络、不同命令集可以灵活组合。来源：[NVMe 2.0 announcement](https://www.businesswire.com/news/home/20210603005088/en/)

可以把三代块存储网络理解为：

```text
FC SAN：专用高性能存储网络
iSCSI：SCSI over TCP/IP
NVMe-oF：NVMe over Fabric
```

iSCSI 让普通 IP 网络承载块存储；NVMe-oF 则让现代 NVMe SSD 的低延迟、高并发能力尽量跨网络延伸。

## 分区存储（ZNS / SMR）：用顺序写换容量与寿命

随着 QLC 闪存和高密 HDD 普及，业界引入了**分区存储**：把盘的地址空间切成一个个 zone，每个 zone 按写指针顺序写，要重写通常得先整体重置。这里主要指主机可感知或主机管理的 zoned 设备；drive-managed SMR 会在盘内隐藏部分顺序写约束。放弃随机覆盖，换来更高容量、更好寿命和更低成本。

- **SMR（叠瓦式磁记录）HDD**：磁道像屋瓦一样部分重叠以提高面密度；host-managed / host-aware SMR 会把 zone 约束暴露给主机，相关接口是 ZBC/ZAC。
- **ZNS（Zoned Namespaces）SSD**：NVMe 的命名空间分区扩展，让主机写入对齐闪存擦除块，大幅减少垃圾回收、预留空间和写放大——这正是 QLC 大规模落地的关键。

来源：[zonedstorage.io - ZNS](https://zonedstorage.io/docs/introduction/zns)、[NVM Express - ZNS](https://nvmexpress.org/specification/nvme-zoned-namespaces-zns-command-set-specification/)

## 计算型存储：让计算靠近数据

传统模型里，数据总要搬到 CPU 才能处理。**计算型存储**（SNIA 标准）反其道而行：把计算下推到盘内或近盘，在数据所在处做过滤、扫描、压缩、检索，只把结果回传主机，避免把整份数据搬过总线。

SNIA 的《Computational Storage Architecture and Programming Model》定义了带计算能力的存储设备（CSx），包括自带存储 + 计算引擎的计算型存储盘（CSD）、只有计算引擎的处理器（CSP）等。对扫描密集的分析和 AI 预处理场景，这能显著减少数据搬运。来源：[SNIA - What is Computational Storage](https://www.snia.org/education/what-is-computational-storage)

## 磁带没有消失：归档和冷数据仍然需要低成本介质

很多人以为磁带已经过时，但在企业归档、冷数据、离线备份、勒索软件防护、合规保存等场景里，磁带仍然有生命力——尤其是它天然支持**空气隔离**。

不过磁带的演进也不是一帆风顺。LTO 是 HPE、IBM、Quantum 持续推进的开放磁带格式。**LTO-10 是一个值得记录的“演进受阻”案例**：它于 **2025 年 6 月**先以 **30TB 原生（75TB 压缩，2.5:1）** 发布，低于此前路线图预期，也打破了 LTO 长期“每代容量翻倍”的惯例；直到 **2025 年 11 月** 才补充宣布 **40TB 原生（100TB 压缩）** 的高密规格，并预期 **2026 年第一季度** 出货。来源：[Fujifilm - LTO-10 发布（2025-06）](https://www.fujifilm.com/us/en/news/data-storage/fujifilm-launches-lto-ultrium-10-data-cartridge)、[LTO 40TB 公告（2025-11，含 Q1 2026 出货）](https://www.lto.org/2025/11/lto-program-announces-new-40-tb-lto-10-cartridge-specifications-and-refreshes-its-roadmap-for-ultra-high-density-ai-ready-archival-storage/)、[Blocks & Files](https://blocksandfiles.com/2025/11/13/lto10-upgrade/)

因此，未来不是“SSD 替代所有东西”，而是更加明显的分层：

```text
极热数据：HBM / DRAM
热数据：NVMe SSD
温数据：SSD / HDD
冷数据：对象存储 / 大容量 HDD
归档数据：磁带 / 离线介质
```

## 介质演进：HDD、HAMR、QLC、持久内存与 CXL

存储介质没有停止演进。

HDD 仍是大容量冷/温数据的重要介质。Seagate 称 HAMR 通过提高面密度来提升容量、降低能耗和总拥有成本。来源：[Seagate - HAMR](https://www.seagate.com/blog/the-future-of-data-storage-technology-why-hamr-is-the-new-standard-for-hard-drives/)

SSD 则从 SLC/MLC/TLC 走向 QLC 等更高密度形态，在性能、寿命、成本之间继续折中，并靠 NVMe 和 ZNS 释放接口与寿命潜力。

**持久内存这条线则经历了一次完整的兴衰，很值得记录。** Intel/Micron 的 3D XPoint（产品名 Optane）是介于 DRAM 和 NAND 之间的持久内存层：可按字节寻址、断电不丢、延迟和寿命远好于 NAND，但比 DRAM 慢、便宜，曾以 Optane SSD 和插内存槽的 Optane PMem 两种形态出货。然而 Micron 2021 年退出，Intel 也在 2022 年第二季度启动 Optane memory business 的 wind-down，并表示停止未来 Optane 产品开发，这条技术线就此终止。来源：[Intel Q2 2022 财报](https://www.intc.com/news-events/press-releases/detail/1563/intel-reports-second-quarter-2022-financial-results)、[Intel Optane Business Update](https://www.intel.com/content/www/us/en/support/articles/000091826/memory-and-storage.html)

接棒“DRAM 之下那一层”的是 **CXL**。CXL 基于 PCIe，提供缓存一致的设备互联，可做内存扩展、池化和分层。它的版本史与底层 PCIe 代际同步推进：

| 版本 | 时间 | 底层 PCIe | 速率 | 关键能力 |
| --- | --- | --- | --- | --- |
| CXL 1.0/1.1 | 2019 | PCIe 5.0 | 32 GT/s | 设备一致性、设备内存访问 |
| CXL 2.0 | 2020 | PCIe 5.0 | 32 GT/s | switching、内存池化、持久内存 |
| CXL 3.0 | 2022 | PCIe 6.0 | 64 GT/s | fabric、内存共享 |
| CXL 3.1 | 2023 | PCIe 6.x | 64 GT/s | fabric 增强 |
| CXL 4.0 | 2025-11 | PCIe 7.0 | 128 GT/s | 多机架池化、链路加宽 |

来源：[CXL 4.0 白皮书（2025-11）](https://computeexpresslink.org/wp-content/uploads/2025/11/CXL_4.0-White-Paper_FINAL.pdf)、[CXL 3.1 发布](https://www.businesswire.com/news/home/20231114332690/en/)

CXL 内部分三个子协议：CXL.io（设备发现与 DMA）、CXL.cache（设备一致地缓存主机内存）、CXL.mem（主机用 load/store 访问设备内存）；设备分 Type 1/2/3（无自带内存的加速器 / 自带内存的加速器 / 内存扩展与池化设备）。

需要注意：CXL 内存通常仍是**易失** DRAM，它接的是 Optane 留下的“分层与容量扩展”角色，而非“按字节寻址的持久性”。长期看，CXL 的意义不只是“更快”，而是把内存扩展、池化、异构设备共享和可组合基础设施带进数据中心。

## 数据增长：为什么存储问题越来越重要

IDC/Seagate 的 Data Age 白皮书曾在 **2018 年预测**：全球数据圈（Global DataSphere）将从 2018 年的 33ZB 增长到 2025 年的 175ZB。来源：[Seagate/IDC - Data Age 2025](https://www.seagate.com/files/www-content/our-story/trends/files/idc-seagate-dataage-whitepaper.pdf)

这是一个偏早期的预测，需谨慎看待。据引用 IDC Global DataSphere 预测的公开资料，更新后的口径约为 **2023 年 132ZB、到 2028 年约 394ZB**，主要由 AI/ML 和云驱动（IDC 原始报告需订阅，此处采用可公开访问的二级来源）。来源：[Embedded Computing Design（引 IDC：132ZB→394ZB）](https://embeddedcomputing.com/technology/storage/beyond-the-hype-efficient-data-storage-at-scale)、[IDC Global DataSphere Forecast 2025-2029 报告页](https://my.idc.com/getdoc.jsp?containerId=US53363625)

无论具体数字如何，它反映了一个现实：数据增长速度远超传统人工管理能力。这推动存储系统从“容量设备”变成“数据管理系统”：自动分层、生命周期管理、数据治理、元数据检索、跨云迁移、安全审计、勒索软件防护、成本可视化。

## 数据湖仓与开放表格式

当数据都堆进了对象存储，下一个问题是“怎么像数据库一样可靠地用它”。**数据湖仓（lakehouse）** 就是答案：在对象存储里的 Parquet 等文件之上，叠一层**开放表格式**的元数据，把“一堆文件”变成“一张可事务操作的表”。

Apache Iceberg、Delta Lake、Apache Hudi 是三种主流表格式，都补上了裸对象存储缺的数据库能力：**ACID 事务**（并发读写不会看到半成品）、**模式演进**（安全增删改列）、**时间旅行**（查询某个历史快照）。三者各有侧重：Iceberg 引擎中立、适合多引擎大规模分析；Delta Lake 基于事务日志、与 Spark/Databricks 深度整合；Hudi 强在 upsert 和增量处理。

它之所以成为现代分析和 AI 的数据层，是因为让 Spark、Trino、Flink、Snowflake 乃至 AI 管线都能在 **S3 上的同一份数据** 上可靠协作，且不锁定单一厂商。来源：[min.io - 开放表格式](https://blog.min.io/architects-guide-to-open-table-formats-and-object-storage/)、[Dremio - Iceberg/Delta/Hudi](https://www.dremio.com/blog/exploring-the-architecture-of-apache-iceberg-delta-lake-and-apache-hudi/)

## AI 时代：存储的新压力

AI 对存储提出了不同于传统数据库和虚拟化的需求：训练数据集规模大；小文件、图片、视频、文本混合；checkpoint 写入频繁；GPU 需要高吞吐、低抖动的数据供给；推理需要低延迟读取模型、特征、向量和上下文；数据还要版本化、可追踪、可治理。

SNIA SDC 2025 的材料把 AI 数据中心存储分成多层：HBM、DRAM、本地 NVMe、远程性能存储、远程容量存储，并讨论 GPU 发起 I/O、GPUDirect Storage、SSD IOPS 等问题。来源：[SNIA SDC25 - Storage Devices in AI Data Center](https://www.snia.org/sites/default/files/2025-10/SNIA-SDC25-Haratsch-Storage-Devices-AI-Data-Center.pdf)

AI 让存储系统同时面对两个极端：

```text
一端是海量容量：数据湖、对象存储、冷数据
一端是极致性能：GPU、HBM、NVMe、CXL、并行文件系统
```

这一节里有几个当前最热的工程焦点，值得单独点名：

- **为什么训练热路径通常不能只靠裸对象存储**：很多管线确实把对象存储当作**源数据层**，再用缓存、FUSE 挂载、数据加载器、并行文件系统或本地 NVMe 来加速；但作为直接喂 GPU 的**高性能热路径**，裸对象存储往往不够——它以对象 API 和 HTTP 请求为中心，即使支持 range GET、multipart upload 等能力，也不提供传统 POSIX 文件系统的低延迟元数据操作和随机小 I/O 语义，而训练框架往往依赖 POSIX 语义和海量小文件的元数据性能。承担热路径的常是**并行文件系统**：Lustre、IBM Storage Scale（原 GPFS）、WekaFS、VAST、DAOS 等，靠单一命名空间、多节点条带、高 IOPS 元数据、预取与缓存把数据塞给 GPU。来源：[Google Cloud - 并行文件系统](https://docs.cloud.google.com/architecture/parallel-file-systems-for-hpc)、[DAOS 官方文档](https://docs.daos.io/)
- **GPUDirect Storage（GDS）**：在存储（本地 NVMe 或 NVMe-oF）和 GPU 显存之间建立直达数据路径，绕过 CPU 内存里的 bounce buffer 中转，缩短 I/O 路径、提升带宽、释放 CPU。来源：[NVIDIA - GPUDirect Storage](https://developer.nvidia.com/gpudirect-storage)
- **checkpoint 写风暴**：万卡训练定期把 TB 级状态同步存盘，会让所有 GPU 一起停下来刷数据，造成训练停顿；业界转向**异步 / 分布式 checkpoint**（如 Gemini 把检查点先写进主机内存、DataStates-LLM 做惰性异步重叠）。来源：[Gemini - SOSP 2023](https://www.cs.rice.edu/~eugeneng/papers/SOSP23.pdf)
- **向量存储**：RAG 把文本/图像编码成 embedding，连同元数据存进向量库，并建 HNSW 等近似最近邻索引以快速“找最相似的 k 个”。代表系统有 FAISS、Milvus、pgvector、Pinecone。来源：[Milvus 文档](https://milvus.io/docs)

IBM 对 AI Storage 的定义也强调，AI 存储要支持快速访问文档、照片和非结构化数据，服务于 NLP、图像识别等工作流。来源：[IBM - What is AI Storage](https://www.ibm.com/think/topics/ai-storage)

未来的 AI 存储不是单一产品，而是一个数据管线：

```text
采集 -> 清洗 -> 标注 -> 数据湖仓 -> 训练缓存 -> checkpoint -> 模型仓库 -> 推理上下文/向量
```

## 管理演进：从设备管理到 API 化管理

早期存储管理是设备级的：磁盘、RAID、卷、LUN、阵列。

云和大规模数据中心需要更统一的 API。SNIA Swordfish 规范用 RESTful 接口语义和标准数据模型，为存储及相关数据服务提供可扩展管理接口；DMTF 也指出 Redfish 与 Swordfish 可以一起覆盖服务器、存储、fabric、NVMe/NVMe-oF 环境。来源：[SNIA Swordfish](https://www.snia.org/tech_activities/standards/curr_standards/swordfish)

这说明存储管理正在从厂商私有工具走向：REST API、标准数据模型、自动化运维、云管平台集成、Infrastructure as Code、Kubernetes CSI。

## 未来演进方向

![存储未来演进五条主线](storage-future-directions.svg)

把前面的线索收敛成五条主线。

### 主线一：分层存储与自动生命周期治理

HDD、SSD、对象存储、磁带会继续分层共存——HBM/DRAM/CXL、NVMe、QLC/HDD、对象、磁带各守一层。关键不是“谁干掉谁”，而是**自动把数据放到合适层级**，并把生命周期与治理一起做起来：归属、能否用于训练、是否含敏感信息、保留期、可审计、跨云迁移、防勒索。存储正从“回答数据在哪里”转向“回答数据该怎么被管理”。

### 主线二：高性能块存储与 NVMe-oF

iSCSI 仍适合通用 IP 网络块存储；NVMe-oF 会在需要低延迟、高并发、高吞吐的数据库、虚拟化和 AI 场景继续扩展，并越来越多地由 DPU 卸载；ZNS 则继续改善 QLC 的寿命与经济性。

### 主线三：对象存储 + 表格式 + 数据湖仓

S3 API 已是事实标准。越来越多系统会用“对象存储 + Iceberg/Delta/Hudi 表格式”组织数据湖仓、训练数据、日志、备份和归档，让一份数据被多引擎可靠协作，又不锁定单一厂商。

### 主线四：AI 数据路径——并行文件系统、缓存、GDS、DPU

喂饱 GPU 的热路径会围绕几件事演进：用并行文件系统（Lustre / Storage Scale / WekaFS / VAST / DAOS）+ 缓存 / 预取承载 POSIX 热数据；用 GPUDirect Storage 让数据直达显存、绕过 CPU 中转；用 DPU / IPU（如 NVIDIA BlueField）把存储 / 网络栈从主机卸载；并把计算挪到数据附近（计算型存储）以减少搬运。传统的 `SSD -> CPU -> GPU` 正被 `SSD -> GPU`、`Remote NVMe -> GPU`、`Object -> Cache -> GPU` 取代。需要说明的是，新出现的开放 AI 网络标准——scale-up 的 **UALink**（对标 NVLink）与 scale-out 的 **Ultra Ethernet（UEC 1.0，2025-06）**（对标 InfiniBand）——本身面向加速器与集群通信、并非存储协议，但会**间接影响 AI 集群里的数据搬运与存储访问路径**。来源：[NVIDIA - BlueField DPU](https://www.nvidia.com/en-us/networking/products/data-processing-unit/)、[Ultra Ethernet Consortium](https://ultraethernet.org/)、[UALink Consortium](https://ualinkconsortium.org/)

### 主线五：CXL 与可组合基础设施

随着 CXL 4.0（2025-11）把速率推到 128 GT/s 并强化多机架池化，数据中心会把 DRAM、CXL Memory、NVMe 组织成更灵活、可组合的层级，内存不再被单机内存槽束缚。一个新兴落点是 **LLM 推理的 KV cache 分层**：把随上下文暴涨的 KV cache 从昂贵的 GPU 显存外溢到 CXL 内存 / 本地 SSD / 远端，从而低成本地服务更长上下文与更高并发。来源：[CXL-based KV Cache（NeurIPS 2024）](https://mlforsystems.org/assets/papers/neurips2024/paper17.pdf)

## 关键结论

1. 存储发展不是线性替代，而是分层共存。
2. HDD、SSD、磁带、对象存储、NVMe-oF、CXL 各自解决不同问题。
3. NAS、SAN、对象存储代表三种不同访问模型，不应混用概念。
4. RAID、纠删码、GFS、Dynamo、Ceph 推动了“用普通硬件构建可靠存储”的工程路线。
5. ZFS、btrfs、Ceph 等系统进一步强调端到端校验、自修复和对静默损坏的防护。
6. S3 让对象存储成为云时代的基础抽象，表格式（Iceberg/Delta/Hudi）又把它升级成可事务的数据湖仓。
7. NVMe、NVMe-oF、ZNS 让块存储的性能与寿命路径重新设计。
8. 持久内存（Optane）的兴衰说明介质演进有成有败；CXL 接过“DRAM 之下那一层”的角色并走向内存池化。
9. AI 把存储从“后台容量系统”推到“前台数据供给系统”，催生并行文件系统、GPUDirect Storage、checkpoint 优化、KV cache 分层、向量存储、DPU 卸载等一连串热点。
10. 未来存储的核心不只是容量，而是数据生命周期、性能路径和治理能力。
