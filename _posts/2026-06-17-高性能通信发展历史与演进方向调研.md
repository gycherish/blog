## 摘要

如果说存储回答“数据放哪、怎么取”，那高性能通信回答的是**“数据怎么在成千上万个计算单元之间快速搬运”**。在 AI 时代，这个问题被推到了前所未有的高度：**通信开销常常成为万卡大模型训练能否线性扩展的关键瓶颈之一**——算力堆上去了，却可能被 GPU 间的数据同步拖住（计算/通信的具体占比取决于模型结构、并行策略、batch、拓扑、是否计算通信重叠等）。

高性能通信的发展，是一部围绕 **延迟、带宽、CPU 开销、可扩展性、成本、开放性** 不断重新平衡的历史。它一直在回答三个问题：

```text
怎么传得快？（低延迟、高带宽）
怎么少占 CPU？（绕过内核、贴近硬件）
怎么扩得大？（从几十张卡到几十万张卡）
```

其中**最重要的一条 scale-out 主线**贯穿二十年：**绕过操作系统、让网卡直接读写远端内存（RDMA）**，再围绕它解决无损、拥塞、集合通信、拓扑、开放标准这一系列难题；此外还有 HPC 传统通信栈（MPI / UCX）、scale-up 直连（NVLink / UALink）、光互连等并行的几条线。今天整个领域分成两个战场——机柜内把 GPU 直连成“一个大 GPU”的 **scale-up**（NVLink / UALink），和跨机柜组网的 **scale-out**（InfiniBand / 以太网）。

## 历史脉络：以太网与专用网络的二十年拉锯

### 专用 HPC 网络：Myrinet 与 Quadrics

上世纪 90 年代，普通以太网延迟高、CPU 开销大，撑不起科学计算集群。于是出现了一批**专用低延迟互连**。Myricom（成立于 1994 年）推出的 Myrinet 一度是 HPC 主流——2005 年 6 月的 TOP500 里约 28% 的系统在用它、到 2007 年 11 月已骤降到约 3.6%（占比据二级资料，精确数字宜以 TOP500 历史榜单复核），被千兆以太网和 InfiniBand 两面夹击。Quadrics（1996 年成立）的 QsNet 走更极致的低延迟路线，公司于 2009 年关闭。来源：[Myrinet（Wikipedia）](https://en.wikipedia.org/wiki/Myrinet)、[Quadrics（Wikipedia）](https://en.wikipedia.org/wiki/Quadrics_(company))

> 一句话定位：专用网络赢在延迟、输在生态与成本，最终被 InfiniBand 和以太网取代——这正是后面所有“专有 vs 开放”之争的预演。

### InfiniBand：高性能网络的事实标准

1999 年，业界两个竞争设计 **Future I/O**（Compaq/IBM/HP）与 **Next Generation I/O**（Intel 主导）合并，成立 InfiniBand 行业协会（IBTA），2000 年发布 InfiniBand 架构规范 v1.0。IB 定义了一套硬件传输协议，原生支持可靠消息传递与内存语义操作（即 RDMA），此后在 HPC 和高端互连领域占据了约二十年的事实标准地位。来源：[IBTA 官方](https://www.infinibandta.org/about-the-ibta/)、[InfiniBand（Wikipedia）](https://en.wikipedia.org/wiki/InfiniBand)（“约二十年事实标准”为行业普遍表述，非单一量化）

### 以太网上的 RDMA：iWARP 与 RoCE 两条路线

InfiniBand 好，但要单独建一张网、换一套网卡和交换机，贵。于是出现了**让 RDMA 跑在以太网上**的两条竞争路线：

- **iWARP**（internet Wide Area RDMA Protocol）：把 RDMA 分层在 **TCP/IP** 之上，由 IETF 标准化，核心 RFC 5040（RDMAP）于 2007 年 10 月发布。好处是可路由、复用成熟的 TCP 拥塞控制；代价是 TCP 协议栈本身的开销。来源：[RFC 5040（IETF）](https://datatracker.ietf.org/doc/rfc5040/)
- **RoCE**（RDMA over Converged Ethernet）：把 RDMA 直接放在以太网上。**RoCE v1（2010，IBTA）** 是纯二层、不可路由；**RoCE v2（2014，IBTA）** 把传输封进 UDP，改用标准 IP 路由，从而能跨三层网络部署。来源：[IBTA RoCEv2 发布](https://www.infinibandta.org/infiniband-trade-association-releases-updated-specification-for-remote-direct-memory-access-over-converged-ethernet-roce/)

历史最终选择了 RoCE 这条路（以太网生态更大），但它带来一个十年难题：**以太网默认是有损的，而传统 RoCE/RNIC 部署通常依赖近似无损网络**——这是后面整章“拥塞控制”的根源。

### Omni-Path：一段兴衰与复活

Intel 在 2015 年底推出 Omni-Path 架构（OPA），2016 年随 Knights Landing 落地 100Gb 的 OPA100，试图挑战 InfiniBand。但 2019 年 7 月 Intel 宣布停止开发、砍掉下一代 OPA200。来源：[Intel pulls the plug on Omni-Path（Network World）](https://www.networkworld.com/article/967727/intel-pulls-the-plug-on-omni-path-networking-fabric-architecture.html)

故事没完：2020 年 9 月 Intel 把 Omni-Path 剥离给新公司 **Cornelis Networks**，后者面向 AI/HPC 重构，于 2025 年 6 月发布 400G 的 **CN5000** 家族并开始出货。来源：[Cornelis CN5000 发布](https://www.cornelis.com/stories/cornelis-launches-cn5000-industry-leading-ai-and-hpc-scale-out-network)（其“比 InfiniBand 快 2x”等为厂商自报基准）

在当代专用/增强型 HPC fabric 里，**HPE Cray 的 Slingshot**（一种增强型、兼容以太网的 HPC 互连）是必须提到的代表——它正是 Frontier、Aurora、El Capitan 这三台顶级超算的网络。可以说专用 HPC fabric 的代表已从 Myrinet/QsNet 换到了 Slingshot。来源：[HPE Slingshot 互连（HPE 官方）](https://www.hpe.com/uk/en/compute/hpc/slingshot-interconnect.html)、[Ethernet's exascale victory（SDxCentral）](https://www.sdxcentral.com/analysis/ethernets-exascale-victory-hpes-slingshot-continues-to-conquer-the-exascale-interconnect-battle/)（后者为二级）

### AI 时代的再变革（2023+）

2023 年 7 月，AMD、Arista、Broadcom、Cisco、Eviden、HPE、Intel、Meta、Microsoft 联合成立 **Ultra Ethernet Consortium（UEC）**，要为 AI/HPC 从头重构以太网通信栈。来源：[UEC 成立（Linux Foundation）](https://www.linuxfoundation.org/press/announcing-ultra-ethernet-consortium-uec)

驱动力是 LLM 训练：万卡集群催生了专用的 AI **后端（back-end）网络**，以太网阵营借机回潮挑战 InfiniBand。据 Dell'Oro 估算，2023 年 InfiniBand 在 AI 后端网络约占八成份额——但这是分析机构数字、各家有出入，应降调看待。来源：[Inside Spectrum-X（SDxCentral，引 Dell'Oro）](https://www.sdxcentral.com/analysis/inside-spectrum-x-nvidias-ethernet-networking-platform/)

NVIDIA 的应对是**两面下注**：用 Quantum InfiniBand 守住训练 fabric，又推出 **Spectrum-X** 以太网平台（把 IB 的自适应路由、拥塞控制移植到以太网）抢以太网那一侧。来源：[NVIDIA Networking 官方](https://www.nvidia.com/en-us/networking/)

## 核心机制

### RDMA：绕过内核、零拷贝、直读远端内存

![RDMA、RoCE 与 PFC 机制示意](high-perf-comm-rdma-roce-pfc.svg)

RDMA 相对 TCP 的性能优势来自一句话：**把网络协议整个放到网卡上，绕过主机内核协议栈（kernel bypass）**，数据在预先注册的内存缓冲区里零拷贝进出，从而大幅降低 CPU 开销与延迟。需要厘清的是：**“假设网络无损”是 RoCE 这条路线的传统取舍，并非 RDMA 的本质要求**——iWARP 就跑在 TCP 之上，IRN 也专门论证了 RDMA 可以不依赖无损网络（见下文「拥塞控制」）。来源：[DCQCN 论文（SIGCOMM'15）](https://conferences.sigcomm.org/sigcomm/2015/pdf/papers/p523.pdf)

编程模型上，RDMA 的接口叫 **verbs**（动词：read / write / send / recv），通信端点是**队列对 QP**（发送队列 + 接收队列，配完成队列 CQ）。其中**单边操作**（read/write）可以直接读写远端内存，对端 CPU 完全无感——这是它延迟低、不打扰对端的关键。

### 无损网络与 PFC 的代价

RoCE 要跑好，需要无损网络，业界靠 **PFC（优先级流控）** 实现：链路快满时让上游暂停发包。但 PFC 是**按优先级/队列在链路级做 pause 的机制、不区分具体流**（粒度远粗于单条流），于是一处拥塞会**沿着暂停信号扩散**，引发队头阻塞、拥塞扩散，甚至**死锁**。来源：[DCQCN 论文](https://conferences.sigcomm.org/sigcomm/2015/pdf/papers/p523.pdf)、[IRN 论文](https://radhikam.web.illinois.edu/irn.pdf)

微软是最早大规模在商用以太网上跑 RoCEv2 的厂商之一，实践中踩到三个真实的坑：**PFC 诱发死锁、RDMA 传输 livelock、网卡 PFC pause 风暴**；并为突破 VLAN 边界设计了基于 DSCP 的 PFC。来源：[RDMA over Commodity Ethernet at Scale（微软）](https://www.microsoft.com/en-us/research/publication/rdma-commodity-ethernet-scale/)

### 拥塞控制的演进：一条为 AI 越改越精细的主线

“PFC 太粗”逼出了一长串更聪明的拥塞控制方案：

- **DCQCN**（SIGCOMM'15，微软）：基于 **ECN** 的端到端速率控制，为 RoCEv2 在 IP 路由数据中心部署而生，长期是 RoCE 的事实标准拥塞控制方案。来源：[DCQCN](https://conferences.sigcomm.org/sigcomm/2015/pdf/papers/p523.pdf)
- **TIMELY**（Google）：改用主机测的 **RTT** 当拥塞信号，不依赖交换机反馈；评测中把 99 分位尾延迟降了约 9–13 倍。来源：[TIMELY（Google Research）](https://research.google/pubs/timely-rtt-based-congestion-control-for-the-datacenter/)
- **HPCC**（阿里）：用**网内遥测 INT** 拿到精确的每链路负载，**通常一步**就算出准确速率，而非反复迭代逼近；比 DCQCN/TIMELY 的流完成时间缩短最多 95%。来源：[HPCC 论文](https://liyuliang001.github.io/publications/hpcc.pdf)
- **IRN**（学术）：更激进——证明 **PFC 并不是 RDMA 的本质需求**，而是当前网卡设计的产物；改用选择性重传 + BDP 流控后，**无 PFC 反而比有 PFC 的 RoCE 快 6–83%**。来源：[IRN 论文](https://radhikam.web.illinois.edu/irn.pdf)

传统方案（DCQCN/TIMELY）的根本局限被 HPCC 论文点破：反馈粗、收敛慢、队列必然先堆起来才反应、调参复杂（**DCQCN 有 15 个旋钮**）。来源：[HPCC 论文](https://liyuliang001.github.io/publications/hpcc.pdf)

到了万卡 AI 规模，连**“传输层到底还要不要做拥塞控制”**这个老共识都被重新审视（拥塞控制可以做在链路级 PFC、网卡的传输层、或集合通信库三个层次；这里指网卡传输层那一层）。据 Meta 在 SIGCOMM 2024 的论文：他们的 RoCE 集群**设计可达 3.2 万 GPU**（已实跑 2.4 万卡集群，其中 1.6 万卡训过一个大号 Llama 3）；在 400G 集群上**干脆放弃了 DCQCN**，仅用 PFC + 集合通信库层面的控制就稳定跑了一年多；并发现默认 **ECMP 路由对 AI 流量表现很差**（流少而大、“大象流”碰撞，曾把训练拖慢 30%+），于是演进出 **Enhanced-ECMP（叠加 QP 扩展，AllReduce 提升约 40%）与集中式流量工程（TE）的组合**——二者按部署规模分工，并非 TE 取代一切。来源：[RDMA over Ethernet for Distributed AI Training at Meta Scale（SIGCOMM 2024）](https://cs.stanford.edu/~keithw/sigcomm2024/sigcomm24-final246-acmpaginated.pdf)（Meta 对自家生产网络的自述）

### 集合通信：训练真正的“网络负载”

![集合通信与拓扑示意](high-perf-comm-collectives-topology.svg)

大模型训练的网络流量，绝大多数是**集合通信**：把每张卡的梯度归约同步。核心原语有 all-reduce、all-gather、reduce-scatter、broadcast，其中 **all-reduce 的常见实现可分解为 reduce-scatter + all-gather**。

- **Ring（环）all-reduce**：每条链路都跑满，带宽最优，但延迟随 GPU 数线性增长；
- **Tree（树）all-reduce**：对数级延迟，规模大、消息小时更优。

NCCL 会按消息大小和拓扑自动在两者间选。来源：[Demystifying NCCL（arXiv）](https://arxiv.org/pdf/2507.04786)。主流库：**NCCL**（NVIDIA，CUDA 事实标准）、**RCCL**（AMD，对 NCCL API 兼容）、**MSCCL**（微软，可自定义按拓扑/消息大小调优的集合算法）。来源：[RCCL（AMD 文档）](https://rocm.docs.amd.com/projects/rccl/en/docs-6.0.2/allapi.html)、[MSCCL（arXiv preprint）](https://arxiv.org/html/2504.09014v4)

### 软件抽象层：MPI 与 verbs 生态

硬件 fabric 之上还需要一层**通信软件栈**，否则应用没法直接对着网卡编程。HPC 世界几十年的事实标准是 **MPI（Message Passing Interface）**——定义进程间消息传递与集合通信的标准接口，科学计算据此编写、底层由各家 fabric 实现。RDMA 这一侧则有 **OpenFabrics 的 verbs 接口**，以及在其上做统一抽象的 **UCX（Unified Communication X）** 和 **libfabric（OFI）**：它们向上主要服务 MPI / PGAS / 通信库，向下适配 InfiniBand、RoCE、iWARP、Slingshot 等不同传输；NCCL 则主要走自己的 net plugin / transport 抽象，也可通过 OFI/UCX 等插件接入部分 fabric。

可以这样理解分层：**MPI / NCCL（编程与集合语义）→ UCX / libfabric / verbs（传输抽象）→ RDMA 网卡与 fabric（硬件）**。AI 时代的 NCCL/RCCL 是“为 GPU 集合通信特化的同类层”，但传统 HPC 栈（MPI + UCX/libfabric）至今仍是超算与科学计算的主干。

### 拓扑与路由：为集合通信而设计

- **fat-tree / Clos**：多级无阻塞、任意点对点全带宽，LLM 训练 scale-out 的默认拓扑。
- **rail-optimized（轨道优化）**：每张 GPU 的网卡（“rail”）接到专属的 rail 交换机——同 rail 的流量只跨一跳，正好契合同步集合通信，跨 rail 再走机内 NVLink。来源：[GPU 集群拓扑（Introl）](https://introl.com/blog/gpu-cluster-network-topology-fat-tree-dragonfly-rail-optimized-2025)、[LLM 基础设施综述（arXiv）](https://arxiv.org/pdf/2407.20018)
- **dragonfly**：组内密连、组间稀疏长链，超大规模下省长线缆与成本。

路由上，**为什么 ECMP 不适合 AI**：AI 流量“熵低”（流少而大），ECMP 按整条流哈希到单条路径，大象流容易**撞在同一条链路**上、白白浪费等分带宽。解法是**逐包喷洒（packet spraying）+ 实时自适应路由**——UEC、NVIDIA Spectrum-X、Broadcom 都在往这个方向走。来源：[Elephant flows（Kentik）](https://www.kentik.com/blog/elephant-flows-the-hidden-heavyweights-of-ai-data-center-networks/)、[UEC 规范更新](https://ultraethernet.org/ultra-ethernet-specification-update/)、[Spectrum-X 自适应路由（NVIDIA）](https://www.nvidia.com/en-us/networking/spectrumx/)

## 现状技术版图：scale-up 与 scale-out 两个战场

![AI 网络的 scale-up 与 scale-out 两个战场](high-perf-comm-scale-up-out.svg)

AI 集群网络分成两层，解决的问题完全不同：

```text
scale-up（纵向，机柜内）：把几十张 GPU 用超高带宽直连成“一个大 GPU”
    代表：NVLink / NVSwitch（专有）、UALink（开放）

scale-out（横向，跨机柜/集群）：把成千上万张卡连成一张大网，主要承载跨节点集合通信（部分架构中也与存储数据路径交叉）
    代表：InfiniBand（NVIDIA Quantum）、以太网/RoCE（NVIDIA Spectrum-X、UEC）
```

### scale-up：NVLink / NVSwitch 与开放挑战者 UALink

**NVLink** 是 NVIDIA 的 GPU 直连互连，带宽逐代翻番：NVLink 1（2016，Pascal）约 160 GB/s/GPU → … → **NVLink 4（2022，Hopper）900 GB/s → NVLink 5（2024，Blackwell）1.8 TB/s/GPU**（gen 1–3 数字取自二级资料，按约值看）。**截至 2026，NVIDIA 已把 NVLink 6 / Vera Rubin 放上主线：3.6 TB/s/GPU、Vera Rubin NVL72 聚合 260 TB/s（路线图产品，尚未规模出货）。** 来源：[NVLink（NVIDIA 官方）](https://www.nvidia.com/en-us/data-center/nvlink/)

**NVSwitch** 是 NVLink 的交换芯片，把机柜内 GPU 连成非阻塞全互联。**GB200 NVL72** 把 72 张 Blackwell GPU 放进一个机柜、组成**单一 NVLink 域**，总计 130 TB/s 全互联带宽、约 13.4 TB HBM3E GPU 显存（另有 17 TB Grace CPU 内存；NVIDIA 有时把两者合并称约 30 TB 快速内存，但这并非单一的 GPU 统一显存），对外像“一个大 GPU”；单 NVLink 域最大可扩到 576 张。来源：[GB200 NVL72（NVIDIA 官方）](https://www.nvidia.com/en-us/data-center/gb200-nvl72/)

2025 年 NVIDIA 又推出 **NVLink Fusion**：以 IP/chiplet 形式**允许第三方 CPU/ASIC 接入 NVIDIA 的 rack-scale 架构**。但要注意它**不是开放标准意义上的“开放”**，而是**以 NVIDIA NVLink 生态为中心**的半定制接入（节点里仍需保留 NVIDIA 芯片）。来源：[NVLink Fusion（ServeTheHome）](https://www.servethehome.com/nvidia-announces-nvlink-fusion-bringing-nvlink-to-third-party-cpus-and-accelerators/)

挑战者是 **UALink**：UALink Consortium 于 **2025 年 4 月发布 UALink 200G 1.0** 规范——开放的 scale-up 加速器互连，每链路 800 Gbps，单 pod 最多 1024 个加速器，由 AMD、Intel、Broadcom 等背书，定位是 NVLink 的开放替代。**截至 2026，规范族已扩展到 Common 2.0、200G DL-PL 2.0、Chiplet 1.01、Manageability 1.0 等（新增网内计算 / Chiplet / 管理能力），200G 1.0 仍是历史起点。** 来源：[UALink 1.0 规范概览](https://ualinkconsortium.org/wp-content/uploads/2025/04/UALink-1.0-Specification-Overview_FINAL-1.pdf)、[UALink 规范页](https://ualinkconsortium.org/specification/)

### scale-out：InfiniBand vs 以太网

scale-out 这一层就是“**InfiniBand vs 以太网**”的主战场。NVIDIA 同时卖两边：**Quantum InfiniBand**（延迟和网内计算占优，守最延迟敏感的训练）与 **Spectrum-X 以太网**（把 IB 的技巧搬到以太网）。关于份额：NVIDIA 2026 财年 Q2 的**书面**口径是「网络业务营收 73 亿美元」；CFO 在该季财报电话会（2025-08-27）**口头**称 **Spectrum-X 年化营收超 100 亿美元**（此句未见于书面 CFO commentary，属电话会口径）。更细的 IB vs 以太网季度拆分多为分析机构估算，本文不采信具体百分比。来源：[NVIDIA Q2 FY2026 业绩（官方新闻，书面网络营收 $7.3B）](https://nvidianews.nvidia.com/news/nvidia-announces-financial-results-for-second-quarter-fiscal-2026)、[CFO 书面评论（SEC EX-99.2）](https://www.sec.gov/Archives/edgar/data/1045810/000104581025000207/q2fy26cfocommentary.htm)、[CFO 电话会引述 Spectrum-X $10B（SDxCentral，二级转述）](https://www.sdxcentral.com/news/nvidia-networking-is-booming-but-your-networks-cost-nothing/)

性能上，独立学术评测（HAICGU 集群）给了一个相对中立的参照：**大消息（>256 KiB）上以太网 RoCEv2 与 InfiniBand 差距在 4% 以内**，但**小消息（<512 B）InfiniBand 最多快 1.6 倍**（RoCEv2 的以太网/IP/UDP 头更大），延迟上 IB 整体仍快约 1.4 倍。来源：[Ethernet vs InfiniBand 评测（arXiv/PDF）](https://danieledesensi.github.io/assets/pdf/2024_EthernetHu.pdf)

### GPUDirect RDMA 与网内计算 SHARP

- **GPUDirect RDMA**：让网卡直接 DMA 读写 GPU 显存、绕过主机 CPU 和系统内存中转，是 NIC↔GPU 的零拷贝路径。来源：[GPUDirect RDMA（NVIDIA CUDA 文档）](https://docs.nvidia.com/cuda/gpudirect-rdma/)
- **SHARP**（Scalable Hierarchical Aggregation and Reduction Protocol）：把集合**归约下推到交换机 ASIC**，数据沿聚合树在网络内就地归约，大幅减少跨网搬运、加速 all-reduce；从 InfiniBand EDR 一路演进到 NDR/XDR，Blackwell 的 NVSwitch 也内嵌了 SHARP。NVIDIA 称可把 all-reduce 有效带宽约翻倍（厂商口径）。来源：[SHARP（NVIDIA 文档）](https://docs.nvidia.com/networking/display/ibdiagnetusermanualv219/nvidia+scalable+hierarchical+aggregation+and+reduction+protocol+(sharp)+support)

### 光互联与共封装光学（CPO）

铜缆在带宽和距离上快到头了，光进铜退。**CPO（共封装光学）** 把光器件直接贴到交换 ASIC 旁，省功耗、提密度与可靠性。NVIDIA 在 2025 年 3 月 GTC 发布 **Quantum-X / Spectrum-X Photonics** CPO 交换机（144×800Gb/s，号称 3.5x 功效）；Broadcom 在 2025 年 10 月推出 102.4T 的 **Tomahawk 6 “Davisson”** CPO 交换机。来源：[NVIDIA CPO 交换机（NVIDIA 新闻）](https://nvidianews.nvidia.com/news/nvidia-spectrum-x-co-packaged-optics-networking-switches-ai-factories)、[Broadcom Tomahawk 6 Davisson](https://investors.broadcom.com/news-releases/news-release-details/broadcom-announces-tomahawkr-6-davisson-industrys-first-1024)

不过**规模落地时点有争议**：SemiAnalysis（2026 年 6 月）认为大规模 CPO 要推迟到 2028–2029、近期仍是可插拔光模块主导；NVIDIA 则公开反驳称今年就会放量。本文两面并陈。来源：[CPO 推迟报道（Seeking Alpha）](https://seekingalpha.com/news/4601927-applied-optoelectronics-leads-networking-stocks-down-following-report-on-cpo-rollout-delay)、[反驳观点](https://globalsemiresearch.substack.com/p/co-packaged-optics-is-not-delayed)。另一条路线是 Google 在 TPU pod 里用的**光路交换 OCS**（Project Apollo，用 MEMS 反光镜替掉电交换 spine）。来源：[Mission Apollo（Google，arXiv）](https://arxiv.org/pdf/2208.10041)

## 当前热点（2024–2026）

### Ultra Ethernet（UEC 1.0）

UEC 于 **2025 年 6 月 11 日**发布 **规范 1.0**（Linux Foundation 旗下），是一套覆盖网卡、交换机、光器件、线缆的**全栈**以太网方案。核心是新传输层 **UET（Ultra Ethernet Transport）**：现代化的 RDMA-over-Ethernet、**多路径逐包喷洒**（避开 ECMP 碰撞）、为 AI/HPC 调优的拥塞控制（发送端自适应窗口 + 接收端信用），目标是在**开放以太网**上拿到 InfiniBand 级的表现。来源：[UEC 1.0 发布（Linux Foundation）](https://www.linuxfoundation.org/press/uec-launches-spec-1.0)、[UEC 规范更新](https://ultraethernet.org/ultra-ethernet-specification-update/)（“InfiniBand 级”是定位话术，非官方基准）。**版本状态**：1.0 是 2025-06-11 初版，截至 2026 年初当前推荐版本已到 **UEC 1.0.2（2026-01-28，主要为拥塞控制算法修订与编辑性澄清）**，中间还有 1.0.1（2025-09）。来源：[UEC 1.0.2 规范 PDF（2026-01）](https://ultraethernet.org/wp-content/uploads/sites/20/2026/01/UE-Specification-1.0.2-1.pdf)、[UEC 规范页](https://ultraethernet.org/specification/)。值得一提：NVIDIA 是**后来才以普通会员身份加入** UEC，并非创始成员。来源：[UEC 新增成员（含 NVIDIA）](https://ultraethernet.org/ultra-ethernet-consortium-welcomes-40-new-industry-leaders/)

### UALink 1.0 与“开放 vs 专有”

scale-up 一侧，**UALink 1.0**（2025-04）是对 NVLink 的开放挑战；scale-out 一侧，**UEC** 是对 InfiniBand 的开放挑战。两条战线指向同一个叙事：**开放多厂商标准正在围攻 NVIDIA 的专有互连（NVLink + InfiniBand）**。但格局远未落定——NVIDIA 自己也在以太网侧（Spectrum-X）和开放化（NVLink Fusion）两面下注。

### 超大规模 AI 集群的网络挑战

Meta 的 SIGCOMM 2024 论文是难得的一手实战样本：其 RoCE 设计目标可扩到 3.2 万 GPU，并已在 2.4 万卡级别运行；400G 集群上放弃 DCQCN、依赖 PFC + 集合库控制，并在 ECMP 失效后引入 Enhanced-ECMP 与集中式流量工程（见上文「拥塞控制」）。它说明**到了万卡规模，教科书式的拥塞控制和路由都要重写**。来源：[Meta SIGCOMM 2024](https://cs.stanford.edu/~keithw/sigcomm2024/sigcomm24-final246-acmpaginated.pdf)

### 与存储的交叉

高性能通信和高性能存储在 AI 数据路径上正在融合：**GPUDirect RDMA / GPUDirect Storage** 让数据从远端 NVMe 或网卡**直达 GPU 显存**，绕过 CPU；NVMe-oF 用的也是 RDMA。换句话说，喂饱 GPU 这件事，网络和存储是同一条路径上的两段。来源：[GPUDirect RDMA（NVIDIA）](https://docs.nvidia.com/cuda/gpudirect-rdma/)

## 前瞻方向

![高性能通信未来演进方向](high-perf-comm-future-directions.svg)

### 开放标准围攻专有互连，但胜负未定

UEC（scale-out）与 UALink（scale-up）携开放、多厂商、以太网经济性之势，冲击 NVLink/InfiniBand 的专有壁垒。学术界甚至预测：十年内数据中心与 HPC 会收敛到现代化的以太网方案，取代 TCP 和今天的 RoCE。来源：[重审 RoCE（arXiv）](https://arxiv.org/pdf/2302.03337)。但 NVIDIA 的双线布局 + 软硬件一体生态，使“开放替代”短期更可能是**份额侵蚀**而非颠覆。

### 为 AI 重写的拥塞控制与路由成为标配

逐包喷洒 + 自适应路由 + 网内计算（SHARP）+ 更精确的 CC（INT/HPCC 思路），会从“前沿论文”变成 AI fabric 的默认配置；UEC/UET 已把其中多项写进标准。

### 光进铜退，但节奏有分歧

CPO 与光路交换（OCS）是突破功耗与规模上限的关键路线，长期方向明确；但短期落地节奏（2026 放量还是 2028–2029）业界仍在争。

### scale-up 域持续扩大

单个 NVLink/UALink 域从几十张卡向上千张扩（UALink 1.0 已规划单 pod 最多 1024 个加速器；更大规模则靠多个 pod 的 scale-out 互联）。“一个机柜 = 一个大 GPU”会变成“一排机柜 = 一个大 GPU”。

### 厘清 CXL 与网络的边界

**CXL** 是基于 PCIe 的**缓存一致、load/store 内存语义**互连，干的是机柜内**内存扩展与池化（scale-up 内存）**；而 **RDMA/以太网/InfiniBand** 是**消息传递网络（scale-out）**。两者都让你“访问远端内存”，但 CXL 是硬件一致的直接 load/store，网络是显式收发——别混为一谈。来源：[CXL vs RDMA（ACM）](https://dl.acm.org/doi/10.1145/3634916)

## 关键结论

1. 高性能通信的地基是 **RDMA**：绕过内核（kernel bypass）+ 零拷贝 + 直读远端内存，把 CPU 从数据搬运里解放出来。
2. “**以太网 vs InfiniBand**”是贯穿二十年的主线：iWARP/RoCE 让 RDMA 上了以太网，UEC 想在开放以太网上拿到 InfiniBand 级表现。
3. 以太网“有损”与 **RoCE 传统部署“要无损”** 的矛盾，靠 **PFC** 兜底却带来死锁/扩散，催生了 **DCQCN → TIMELY → HPCC → IRN** 一长串拥塞控制；万卡规模下甚至有人（Meta）连网卡传输层的拥塞控制都不要了。
4. AI 把网络劈成两层：**scale-up**（NVLink/UALink，机柜内 GPU 直连成大 GPU）与 **scale-out**（IB/以太网，跨柜组网）。
5. 真正的网络负载是**集合通信**；为它而生的有 ring/tree all-reduce、rail-optimized 拓扑、网内计算 **SHARP**、逐包喷洒与自适应路由。
6. **开放标准（UEC/UALink）正围攻专有（NVLink/InfiniBand）**，但格局未定，且公开的市场份额数字大多是估算，需降调。
7. **光互联（CPO/OCS）** 是功耗与规模的下一道坎，长期方向明确、落地时点有争议。
8. **CXL（内存语义 scale-up）** 与**网络（消息传递 scale-out）** 是两件事，要分清边界。
9. 通信与存储在 AI 数据路径上融合：**GPUDirect RDMA/Storage**、NVMe-oF 让数据直达 GPU——这正是「AI 存储 + AI 高性能通信」同属一条数据供给链的原因。

### RDMA 与协议

| 术语 | 含义 |
| --- | --- |
| RDMA | Remote Direct Memory Access，远程直接内存访问：网卡直接读写远端内存、绕过对端 CPU |
| verbs | RDMA 的编程接口（动词：read / write / send / recv） |
| QP | Queue Pair，队列对：RDMA 的通信端点（发送队列 + 接收队列） |
| kernel bypass | 内核旁路：网络协议在网卡上处理、绕过主机内核协议栈 |
| zero-copy | 零拷贝：数据在预注册内存缓冲区直接进出，不经多次拷贝 |
| InfiniBand | 一种低延迟、原生支持 RDMA 的高性能交换网络（IBTA 标准） |
| IBTA | InfiniBand Trade Association，InfiniBand 行业协会 |
| iWARP | internet Wide Area RDMA Protocol，跑在 TCP/IP 之上的 RDMA |
| RoCE | RDMA over Converged Ethernet，跑在以太网上的 RDMA（v1 二层、v2 经 UDP 可三层路由） |
| OPA | Omni-Path Architecture，Intel 的专用互连（已转 Cornelis Networks） |

### 无损网络与拥塞控制

| 术语 | 含义 |
| --- | --- |
| PFC | Priority Flow Control，优先级流控：按优先级/traffic class 的链路级 pause 机制，粒度粗于具体流 |
| ECN | Explicit Congestion Notification，显式拥塞通知 |
| HoL blocking | Head-of-Line blocking，队头阻塞 |
| DCQCN | Data Center Quantized Congestion Notification，基于 ECN 的 RoCEv2 拥塞控制 |
| TIMELY | 基于 RTT（往返时延）的数据中心拥塞控制 |
| INT | In-Network Telemetry，网内遥测 |
| HPCC | High Precision Congestion Control，基于 INT 的精确拥塞控制 |
| IRN | Improved RoCE NIC，去 PFC、用选择性重传 + BDP-FC 的改进网卡 |
| ECMP | Equal-Cost Multi-Path，等价多路径（按整条流哈希选路） |
| packet spraying | 逐包喷洒：把一条流的包打散到多条路径，避开 ECMP 碰撞 |

### 集合通信与拓扑

| 术语 | 含义 |
| --- | --- |
| all-reduce / all-gather / reduce-scatter | 集合通信原语；all-reduce 常见实现可分解为 reduce-scatter + all-gather |
| NCCL / RCCL / MSCCL | NVIDIA / AMD / Microsoft 的集合通信库 |
| SHARP | Scalable Hierarchical Aggregation and Reduction Protocol，把集合归约下推到交换机的网内计算 |
| fat-tree / Clos | 多级无阻塞、全带宽的网络拓扑 |
| rail-optimized | 轨道优化：每张 GPU 的网卡接到专属 rail 交换机 |
| dragonfly | 蜻蜓拓扑：组内密连、组间稀疏长链 |
| fabric | 互联的网状交换网络 |

### scale-up / scale-out 与互连

| 术语 | 含义 |
| --- | --- |
| scale-up / scale-out | 纵向扩展（机柜内 GPU 直连成大 GPU）/ 横向扩展（跨柜组网） |
| NVLink / NVSwitch | NVIDIA 的 GPU 直连互连 / NVLink 交换芯片 |
| NVL72 | GB200 NVL72，把 72 张 Blackwell GPU 组成单一 NVLink 域的机柜 |
| NVLink Fusion | 允许第三方 CPU/ASIC 接入 NVIDIA NVLink 生态/机架架构的半定制方案（2025；非开放标准） |
| UALink | Ultra Accelerator Link，开放的 scale-up 加速器互连 |
| GPUDirect RDMA | 网卡直接 DMA 读写 GPU 显存，绕过主机 CPU |
| CXL | Compute Express Link，基于 PCIe 的缓存一致、内存语义互连（内存扩展/池化） |

### AI 网络与光互联

| 术语 | 含义 |
| --- | --- |
| UEC | Ultra Ethernet Consortium / Ultra Ethernet，面向 AI/HPC 的开放以太网全栈 |
